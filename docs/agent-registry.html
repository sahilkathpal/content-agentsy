 Scout
INSurface config — search terms, official_urls, competitor_urls. Pipeline trigger.
OUTraw-buckets.json → Structuring
  • Fires all six data collectors in parallel with per-source concurrency limits.

Analysis
agents/analysis.ts
prompt: analysis.md
 Scout
INsignals.json from Structuring + surface definition.
OUTscout-output.json → Strategist. One ScoutOutput per surface with opportunities[].
  • Clusters signals thematically — each cluster is one potential content opportunity.
  • Generates per cluster: opportunity_id, angle (content hook), friction (user's core problem), outcome (what resolution looks like), proof_assets, signal_ids[].
  • Minimum evidence gate: ≥1 pain signal + ≥1 demand signal + ≥1 freshness signal required. Failures are dropped with a reason.
  • Scores each opportunity 1–5 confidence with a freshness scoring matrix: majority "new" → bonus; majority "recurring" → capped at 3.
  • Suggests format types and distribution channels per opportunity.
  • Attaches a freshness_profile (new/resurfaced/recurring distribution) to each opportunity for Strategist scoring.

Strategist
agents/strategist.ts
prompt: strategist.md
 Strategist
INAll scout-output.json files (all surfaces combined) + optional blog index.
OUTstrategist-output.json → Creator (one packet at a time).
  • Only agent with a cross-surface view — ingests all surfaces' scout outputs simultaneously.
  • Classifies each opportunity into intent mode: M0_RESOLVE (debug/understand) · M1_EVALUATE (compare) · M2_EXECUTE (configure/operate).
  • 5-dimension composite score: demand ×3 · proximity ×2 · proof ×2 · freshness ×1.5 · defensibility ×1.5.
  • Drop logic: confidence <2, composite <25, no minimum evidence, or already in blog index.
  • Creates 1–2 distribution packets per opportunity with: format, primary channel, voice type (engineer / founder_operator / community), grass_role (light / evaluate / execute).
  • Outputs ranked_packets[] sorted by composite score, dropped[] with reasons, strategy_notes[].

Creator
agents/creator.ts
prompt: creator.md
 Creator
INOne distribution packet + scout signals for that surface + grass-context.md + blog index + authority links + source URLs.
OUTcreator-output.json → Publisher + both Derivatives agents.
  • Selects mode-specific structure (M0 / M1 / M2) and enforces 8 GEO optimisation rules: self-contained opening, question-shaped headings, TL;DR block, self-contained paragraphs, comparison table, inline definitions, FAQ section (3–5 Q&As), proof assets woven into narrative.
  • Applies grass_role strictly: light (neutral publisher note only) · evaluate (1–2 honest mentions) · execute (Grass is subject; use grass-context.md).
  • Weaves in ≥2 internal cross-links from blog index and ≥3 external authority links.
  • Links back to community discussion threads when citing user quotes from signals.
  • Response split by ---JSON--- / ---MARKDOWN--- markers — parser extracts metadata and canonical markdown separately.

agents/publisher.ts
Ghost Admin API
 Publisher
INcreator-output.json — canonical markdown + metadata.
OUTpublisher-output.json → Derivatives (canonical URL needed for backlinks).
  • Strips the leading H1 from markdown — Ghost renders the post title separately from body.
  • Parses the FAQ section → generates FAQPage JSON-LD. Generates BlogPosting JSON-LD with author, datePublished, publisher fields.
  • Converts markdown → HTML via marked.js, then wraps in Ghost 5+ Lexical JSON envelope (required by Ghost Admin API).

  • Builds tag set: surface tag + intent_mode + #pipeline + brand tags + strategist format/voice tags.
  • Signs the API request with a JWT (HS256) from the Ghost Admin key. POSTs to /api/admin/posts/ as status="draft".

Scorer
agents/scorer.ts
 Scorecard
INSQLite DB — tables: citations, gsc_site_metrics, gsc_page_metrics, gsc_query_metrics, articles, prompts, competitors. Runs on its own schedule.
OUTscorecard.json → Analyst.
  • Purely computational — no LLM calls. Reads SQLite directly; computes 25 metrics across three layers.
  • Outcome layer (8): domain citations 7d, citations by engine, brand mentions, north-star prompt status, citation position (median/avg), position-weighted SOV per engine, citation coverage %, top competitors.
  • SEO layer (9): site impressions/clicks/CTR 7d, pages with ≥1 impression, branded/non-branded split, rising queries (ratio ≥1.5×), declining queries (ratio ≤0.5×), top pages by clicks, top queries by impressions.
  • Bridge layer — GEO–SEO quadrant: joins citations by slug with GSC page metrics → classifies each article as star / geo_only / seo_only / orphan.
  • Applies configurable threshold checks (min/max per metric); flags breaches for Analyst.
No prompt
No LLM — pure computation
All behaviour is SQL queries + arithmetic. To change what is measured, edit the query logic and threshold config in scorer.ts. No prompt to modify.

Analyst
agents/analyst.ts
prompt: analyst.md
 Analyst
INscorecard.json from Scorer + list of threshold breaches.
OUTanalyst-output.json. Advisory terminal — feeds back to human review → config → next cycle.
  • For each breached threshold: writes explanation, recommended_action, priority (critical / important / watch).
  • Engine concentration risk: flags if any engine >60% (high) or >40% (medium) of citations.
  • Quadrant recovery: orphan → deprecate/re-scout · geo_only → upgrade grass_role · seo_only → add FAQ/GEO structure · star → identify replicable pattern.
  • Mode performance (M0/M1/M2) and format performance correlation with citations and clicks.
  • Rising query analysis → surfaces new content opportunities from GSC data.
  • Outputs: registry_updates[] (promote/demote/drop surfaces), packet_heuristic_updates[] (favor/disfavor format + channel combos), scout_focus_updates[], strategy_notes[] (2–5 plain-language PM insights).
