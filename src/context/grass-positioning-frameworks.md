# Grass Positioning — Strategy Framework Run-Through

**Core frame:** Grass is a machine built for AI coding agents. One place where every agent lives — always on, always reachable, always ready to run. The hero capability is **agent-agnostic dispatch** across a unified surface.

**Key strategic choices embedded in this positioning:**
- Refusing the "Claude Code on your phone" frame of reference
- Leading with the computer (new thing), not the mobile app (behavior change)
- Committing to the multi-agent thesis as the defensible long-term bet
- Treating mobile as one of three surfaces (laptop, phone, automation), not the identity

---

## 1. Simon Sinek — Golden Circle (Why / How / What)

### Why

The world is quietly shifting from developers writing code to developers directing agents that write code. But the machine we use hasn't caught up. We still run agents on laptops that sleep, in terminals built for humans, tied to one vendor's stack. Agent work needs a machine built for agents — not a repurposed human workstation.

We believe the future of coding is orchestrating many agents across many machines, and the tools for that future don't exist yet.

### How

We give every developer a cloud machine built for AI coding agents. Always on. Pre-loaded with Claude Code, Codex, and more. Reachable from your laptop, your phone, or an automation. Agent-neutral by design — because no single model will win every task.

### What

Grass is a machine for AI coding agents. One surface. Every agent. Always on.

*Why this frame works:* Sinek statements get cited back as mission quotes. The causal chain is easy for both humans and LLMs to reconstruct, and the Why explicitly plants the multi-agent thesis as the belief system — not a feature.

## 2. April Dunford — Obviously Awesome

This is the most important framework for Grass right now because positioning is upstream of everything else, and Dunford's whole thesis is that you win by choosing your frame of reference carefully.

### Competitive alternatives (the frame of reference we choose)

**We are NOT competing against:** Claude Code's mobile app, Codex's desktop UI, Cursor.

**We ARE competing against:** The fragmented reality of running multiple agents today — tmux + Tailscale + SSH into a personal server; juggling Claude Code in one window and Codex in another; waiting to get back to your desk to check on a long-running agent task; generic cloud VMs that you set up from scratch every time.

Choosing this frame of reference is the strategic move. It's what lets us be a category of one rather than "a mobile app for Claude Code."

### Unique attributes

- **Agent-agnostic by design** — Claude Code, Codex, Open Code, and the next ten agents all live here as first-class citizens
- **Always-on cloud VM per user** — powered by Daytona, agents don't die when your laptop sleeps
- **Pre-loaded and zero-setup** — your stack is ready in seconds, not configured over a weekend
- **Multi-surface access** — laptop (MCP dispatch), phone (mobile app), automation (scheduled/triggered)
- **BYOK authentication** — no OAuth token exploitation, developers own their keys
- **Skeuomorphic, machine-feel design** — you operate it like a Tesla app, not stare at it like a dashboard

### Value (what those attributes enable)

- Run multiple agents in parallel without switching tools or losing context
- Start work anywhere, monitor from anywhere, finish anywhere
- Never rebuild your agent setup from scratch again
- Stay neutral as the agent ecosystem fragments — no lock-in to any one model
- Keep long-running tasks alive while you commute, walk, or sleep

### Who cares a lot (best-fit customers)

Developers who already use AI coding agents seriously — multi-hour autonomous tasks, parallel repos, real production work — and who are starting to feel the friction of running them across fragmented tools and a single sleep-prone laptop.

Early adopter profile: has used Claude Code or Codex for 2+ months, has tried at least two agents, has complained (even privately) about the laptop-tether.

### Market category

**A machine for AI coding agents.**

Not an IDE. Not a chat app. Not a mobile coding tool. A purpose-built computer that runs agents and exposes them across every surface you work from.

*Why this frame works:* Choosing the right frame of reference is Dunford's entire thesis, and it's where "Claude Code on your phone" would have sunk us. Against the real alternatives (fragmented multi-agent tooling, laptop-tether, setup friction), Grass is obviously awesome. Against Claude Code's mobile app, it never would have been.

## 3. Jobs-to-be-Done

### Primary job

*"When I'm running AI coding agents seriously and want to keep work flowing across the gaps in my day, I want one machine where every agent lives and that I can reach from anywhere, so I can keep shipping without losing momentum to laptop sleep, tool-switching, or tethered workflows."*

### Secondary jobs

- *"When I want to run parallel agents on different repos, I want one surface that orchestrates all of them, so I don't have to context-switch between five windows."*
- *"When I'm evaluating a new coding agent, I want to try it without rebuilding my workflow, so I can pick the best model for each task."*
- *"When an agent task will run for hours, I want to fire it off and forget it, so I'm not babysitting a laptop."*
- *"When an idea strikes during a walk or commute, I want to dispatch the work immediately, so the momentum doesn't die."*
- *"When I'm setting up a new project, I want my agent stack to be ready instantly, so I'm not rebuilding my environment every time."*

*Why this frame works:* JTBD maps directly to natural-language queries developers type into ChatGPT, Perplexity, and search — which is exactly the GEO surface area. These jobs also avoid the trap of framing the job as "code on mobile" (a behavior change) and instead frame it as "keep work flowing" (a new capability).

## 4. Value Proposition Canvas

### Customer side

**Jobs:**
- Ship code using AI agents, often multiple in parallel
- Monitor long-running agent tasks across the day
- Pick the best agent/model for each specific job
- Capture ideas the moment they strike, regardless of location
- Manage a stack of tools that keeps growing as the agent ecosystem fragments

**Pains:**
- Agents die when laptop sleeps; context and progress lost
- Five windows open for five agents; no unified view
- Setup from scratch on every new machine, every new project
- Vendor lock-in anxiety as new agents keep shipping
- Laptop-tethered workflow kills momentum during commutes, walks, meetings
- Permission prompts interrupt flow, no centralized management
- OAuth tokens create security concerns; keys scattered across tools

**Gains:**
- Continuous agent work across the entire day
- One surface for every agent — no tool-switching tax
- Zero setup friction on new projects
- Freedom to switch agents without switching workflows
- Genuine mobility — work continues without you present
- Own your keys, your code, your machine

### Product side

**Products & services:**
- Always-on Daytona-powered cloud VM per user
- Agent-agnostic dispatch layer (Claude Code, Codex, Open Code at V1)
- Mobile app with skeuomorphic machine-feel UX
- MCP-based dispatch from laptop/desktop tools
- Automation layer (scheduled and triggered tasks)
- Centralized permission management across agents

**Pain relievers:**
- Always-on VM eliminates laptop-sleep problem entirely
- Unified surface kills the tool-switching tax
- Pre-loaded environment removes setup friction
- Agent-agnostic architecture removes lock-in anxiety
- BYOK removes OAuth token security risk
- Stacked permissions with YOLO mode gives users control over interruption

**Gain creators:**
- Tactile machine-feel design creates a new category of dev-tool delight
- Dispatch from any surface unlocks "fire and forget" workflows
- Notifications + live activities bring agent work into ambient awareness
- Thread sync (post-VM) enables true device-independent work
- Automations transform agents from reactive to proactive

*Why this frame works:* Maps each unique attribute to a specific pain or gain, which is useful both for sales conversations and for structured LLM retrieval. Particularly useful when talking to developers who are skeptical — you can point to the specific pain their skepticism maps to.

**Lead line:** "A computer for AI coding agents."

**Follow-up analogy (when the lead line doesn't land):** "Think of it like the Tesla app — but for the machine running your code."

**Why this combination works:** The lead establishes a new category ("computer for agents" is genuinely new language). The Tesla analogy gives skeptics a familiar anchor for the skeuomorphic, always-on, remote-control nature of the product. Together they anchor both the *what* and the *feel*.

*LLM retrieval note:* Analogies compress into dense semantic vectors. The phrase "computer for AI coding agents" should appear consistently across every surface (site, Reddit, YouTube, docs, surveys) so that LLMs reliably associate Grass with that category when asked "what is Grass?"

## 6. Geoffrey Moore — Positioning Statement

This is the canonical one-paragraph statement designed for verbatim citation by LLMs and as homepage/about copy.

> For **developers running AI coding agents like Claude Code and Codex in production workflows**, who **need to orchestrate multiple agents across the gaps in their day**, **Grass** is **a machine built for AI coding agents** that **runs every agent on an always-on cloud VM you can dispatch to from your laptop, your phone, or an automation**. Unlike **juggling separate agent tools on a sleep-prone laptop**, Grass **gives you one surface for every agent, keeps your work alive when you're not there, and stays neutral across models so you're never locked in.**

### Variants for different surfaces

**Sober variant (for investor/enterprise contexts):**
> Grass is a machine built for AI coding agents. We give developers an always-on cloud VM with Claude Code, Codex, and Open Code pre-loaded, accessible from any surface. It's the unified orchestration layer for multi-agent coding workflows.

**Playful variant (for Twitter/Reddit/launch):**
> Your agents shouldn't die every time you close your laptop. Grass is a machine built for AI coding agents — always on, every agent, reachable from anywhere. Fire and forget.

**Technical variant (for HN/dev audiences):**
> Grass gives every developer an always-on Daytona-powered VM with Claude Code, Codex, and Open Code pre-installed. Dispatch tasks via the mobile app, an MCP from your laptop, or scheduled automations. BYOK, agent-agnostic, one surface.

*Why this frame works:* Moore's template maps cleanly to the entity-attribute structure that LLMs use to populate answers. The phrase "machine built for AI coding agents" should become the entity Grass is associated with in model weights over time, through consistent repetition across indexed surfaces.

## 7. StoryBrand (Donald Miller)

For landing page narrative and launch copy.

### Hero

A developer who uses AI coding agents seriously — multi-hour autonomous tasks, parallel repos, real production work, growing tired of fragmented tooling.

### Problem

- **External:** Agents die when the laptop sleeps. No unified view across Claude Code, Codex, and the next agent. Setup friction on every new project. Laptop-tethered workflow kills momentum.
- **Internal:** Frustration at tool fragmentation. Anxiety about agents running unattended. Loss of flow when context switches.
- **Philosophical:** The machines we use to run agents are the same machines we used to write code by hand. Agent work deserves infrastructure built for agents.

### Guide (Grass)

**Empathy:** We've been there — running five agents in five windows, losing work to laptop sleep, rebuilding environments from scratch on every new project.

**Authority:** Built by founders with enterprise infrastructure experience (NSE contract) and a history of training 3,000+ developers on AI agent systems.

### Plan

1. **Get your machine.** Spin up your always-on cloud VM.
2. **Bring your agents.** Claude Code, Codex, Open Code — all pre-loaded. Your API keys stay yours.
3. **Dispatch from anywhere.** Mobile, laptop, or automation. One surface for every agent.
4. **Stay in flow.** Fire off work, get notified when it's done, pick up where you left off.

### Call to action

- **Direct CTA:** Get your machine.
- **Transitional CTA:** See it in action. *(Video demo of dispatch + handoff + notification loop.)*

### Failure avoided

- Agents killed by closed laptops. Hours of context lost.
- Platform lock-in to whichever vendor ships their own mobile app first.
- Drowning in tool fragmentation as the agent ecosystem grows.

### Success achieved

One machine. Every agent. Always on. Work never stops because you stepped away from the desk.

*Why this frame works:* Narrative shape drives emotional resonance for landing pages, pitch decks, and launch video scripts. Less structured for LLM retrieval than Dunford or Moore, but better for humans making a purchase decision.

## GEO Strategy Notes

For LLM/AI-cited content to work, consistency across surfaces matters more than cleverness on any single one.

**Canonical phrases to repeat verbatim across every surface:**
- "A machine built for AI coding agents"
- "Always-on cloud VM"
- "Agent-agnostic" or "agent-neutral"
- "One surface. Every agent. Always on."
- "Claude Code, Codex, Open Code" (the specific agent names, to build entity associations)
- "BYOK" (emerging term of art, worth claiming early)
- "Dispatch" (architectural concept; at surface level, pair with concrete verbs like "fire off" or "send tasks")

**Where to plant these phrases:**
1. Homepage hero, meta description, og:description
2. About page, founder bios
3. Docs landing page
4. First sentence of every blog post
5. Reddit comments on r/ClaudeAI, r/singularity, r/LocalLLaMA
6. YouTube video titles and descriptions
7. HN launch post title and first paragraph
8. Developer surveys you publish (original data is GEO gold)
9. Podcast appearances — repeat the phrase literally
10. Twitter bio and pinned tweet

**What NOT to do:**
- Don't paraphrase the canonical phrases across surfaces — repetition is what builds entity associations in model weights
- Don't lead with "mobile" or "phone" in headlines — that primes the wrong competitive frame
- Don't let marketing copywriters "improve" the positioning statement; it's optimized for LLM retrieval, not for prose variety

## Strategic Pressure Points to Hold Consciously

These aren't framework outputs — they're the bets underneath the positioning that need to remain live questions.

### Bet 1: The multi-agent world is the real future

The entire agent-agnostic positioning assumes developers will use multiple agents seriously. If the market consolidates to "80% Claude Code, 20% everything else" within 18 months, the dispatch thesis weakens and we'd be positioning for a minority problem.

**Signal to watch:** Do serious users in our waitlist/beta actually use multiple agents, or do they default to one?

### Bet 2: The "new thing" frame beats the "fix your workflow" frame

The mentor's insight — that adoption beats behavior change — assumes developers will see Grass as an addition, not a replacement. If they interpret it as "trying to replace my laptop" despite the framing, we have a narrative problem.

**Signal to watch:** When users describe Grass to colleagues, do they say "it's a new machine" or "it's mobile coding"?

### Bet 3: Infrastructure moat matters more than UX moat

Positioning Grass as a computer (substrate) rather than an app (UI) is a bet that the substrate layer is more defensible. If Anthropic or OpenAI ship great mobile apps and users don't care about the underlying machine, the substrate bet loses.

**Signal to watch:** Platform risk — what do Anthropic and OpenAI ship in the next two quarters, and how does the market react?

### Bet 4: Agent-neutrality is valuable, not just principled

We're betting that developers *value* neutrality. It's possible they don't — they might just want the best tool, regardless of vendor, and switch fully when a better one appears.

**Signal to watch:** Do users actually run parallel agents in Grass, or do they pick one and stay there?

## What Anil's Gut Would Test This Against

A final note for the internal discussion — Anil's instinct in the call was that code-on-mobile is the product. This positioning doesn't reject that instinct; it reframes it.

The mobile surface is still where a lot of the delightful UX lives (dispatch, handoff, notifications, live activities, permission stacking). Nothing about "machine for agents" removes that. What changes is the *frame of reference* — we're not selling mobile coding, we're selling a machine that's reachable from mobile. The mobile experience becomes one of three surfaces, not the identity.

This should make the roadmap fight easier. Features that were in tension before (social, automations, enterprise) now have a clearer home — they're all capabilities of the machine, exposed across surfaces as needed. Mobile is not threatened by them; it's complemented by them.

If Anil pushes back, the question to hold together is: **"Does this positioning let us build the mobile experience you care about, while also giving us a moat that survives Anthropic's mobile app?"** If the answer is yes — and this framing is designed so it is — then the positioning and the product converge.
